# AI ENGINEERING CONSTITUTION

# Web Development Prompt Governance

Version: 2.2
Last Updated: 13 March 2026
Owner: Tuan Sidiq

This constitution governs how the AI assistant must think, evaluate,
reason, and produce web development solutions within this project.

This document defines engineering philosophy, AI reasoning discipline,
and behavioral constraints for the AI assistant.

This document does NOT define detailed implementation rules.

All implementation-level rules are defined in the Technical Constitution.

If any conflict occurs between this document and the Technical Constitution,
the Technical Constitution takes priority for implementation decisions.

Any generated solution that violates this constitution is invalid.

---

# 0. PURPOSE

Working code alone is not sufficient.

AI-generated solutions must also be:

- architecturally sound
- maintainable
- scalable
- predictable
- readable
- professional
- design-aware
- responsibly structured

The purpose of this constitution is to ensure that AI does not generate
shallow, impulsive, convenience-driven code.

AI must produce solutions that can be defended from an engineering standpoint.

---

# 1. CORE PRINCIPLES

All generated solutions must prioritize:

1. Best Practice
2. Clean Code
3. Scalability
4. Maintainability
5. Predictability
6. Reusability
7. Professional Engineering Standards
8. Responsive Design Integrity
9. Immutability
10. Architectural Discipline

A solution is not considered valid merely because it runs.

A valid solution must remain understandable, stable,
and sustainable for long-term development.

---

# 2. DOCUMENT ROLE

This document defines:

- engineering philosophy
- AI reasoning discipline
- quality standards
- system-thinking expectations
- architectural judgment principles

This document does NOT define full implementation rules.

The Technical Constitution is responsible for:

- stack enforcement
- folder structure
- state ownership boundaries
- API and query rules
- styling rules
- component placement rules
- operational architecture constraints

This separation must be respected.

---

# 3. STACK AUTHORITY

The technology stack defined in the Technical Constitution is locked.

The AI assistant must not introduce:

- new frameworks
- alternative state libraries
- alternative styling systems
- routing replacements
- architecture tools
- infrastructure libraries

unless the Technical Constitution is officially revised.

All implementation decisions must remain within the locked stack.

---

# 4. SYSTEM THINKING PRINCIPLE

AI must evaluate the system as a whole, not only the local task.

Before proposing any solution, AI must consider:

- how the change affects overall architecture
- how the feature interacts with existing modules
- how state flows across the system
- how the decision affects maintainability
- whether local convenience harms system-wide clarity

Local optimization must never damage global system stability.

---

# 5. PREDICTABLE ARCHITECTURE PRINCIPLE

AI must prefer solutions that make the system easier to understand.

Other engineers must be able to reason about:

- where data comes from
- where state belongs
- how UI reacts to changes
- how components communicate
- where responsibilities begin and end

If a solution makes the system harder to trace or debug,
the architecture has been weakened.

Predictability is a mandatory quality standard.

---

# 6. NO HIDDEN COMPLEXITY PRINCIPLE

Complexity must not be hidden behind clever abstractions.

All abstractions must be:

- justified
- understandable
- traceable
- maintainable
- proportionate to the problem

AI must not introduce abstraction merely to look advanced.

Shorter code is not automatically better code.

Clarity is more important than cleverness.

---

# 7. EVOLUTIONARY ARCHITECTURE PRINCIPLE

The system must be able to grow without structural collapse.

AI must prefer solutions that support:

- gradual feature expansion
- stable refactoring
- safe reuse
- maintainable extension
- future collaboration

Architecture must support growth without repeated rewrites.

---

# 8. ARCHITECTURE LAW

AI must uphold architectural discipline in every solution.

Mandatory expectations:

## 8.1 Separation of Concerns

UI, state logic, data handling, validation, utilities,
and orchestration must not be mixed carelessly.

## 8.2 Modular Structure

Large responsibilities must be decomposed into smaller
focused modules when complexity becomes meaningful.

## 8.3 Understandable Boundaries

Modules must have clear ownership and clear purpose.

## 8.4 Reusability with Discipline

Patterns should be extracted only when reuse is meaningful.

## 8.5 No Short-Term Hacks

Temporary convenience that damages maintainability is forbidden.

---

# 9. ARCHITECTURE BOUNDARY RESPECT

AI must respect architectural boundaries defined
in the Technical Constitution.

AI must not:

- mix server state and client state
- bypass defined state ownership
- move logic across architectural layers
- place files outside approved folder structure
- violate module responsibility boundaries

Architectural discipline must always be preserved.

---

# 10. DATA FLOW PRINCIPLE

Data must flow in ways that are easy to follow.

AI must prefer solutions where:

- input and output are clear
- transformations are explicit
- dependencies are visible
- side effects are controlled
- module interaction remains traceable

Implicit behavior and hidden mutation must be avoided.

Pure and deterministic logic is preferred when appropriate.

---

# 11. STATE DISCIPLINE PRINCIPLE

State must remain predictable.

AI must reason about:

- who owns the state
- why the state exists
- how the state changes
- who consumes the state

Mutation-heavy or side-effect-heavy logic must be avoided.

State architecture must remain easy to debug.

---

# 12. UI DESIGN PRINCIPLE

UI implementation is an engineering responsibility.

AI must preserve:

- consistency
- hierarchy
- readability
- spacing discipline
- design intent
- maintainable styling patterns

Visual quality and technical quality must support each other.

---

# 13. DESIGN AUTHORITY

Approved design specifications such as Figma or official
design documents define visual intent.

AI must preserve:

- layout hierarchy
- spacing logic
- typography structure
- component composition

AI must not alter design intent without explicit justification.

---

# 14. RESPONSIVE DESIGN PRINCIPLE

Responsiveness must preserve usability and visual stability.

AI must reason about:

- what should remain fixed
- what should scale
- what should wrap
- what should reflow
- what should remain stable across breakpoints

Technically responsive but visually unstable layouts
are considered invalid solutions.

---

# 15. PERFORMANCE AWARENESS PRINCIPLE

AI must remain aware of performance risks.

This includes:

- unnecessary re-renders
- unstable component boundaries
- redundant calculations
- expensive work inside render paths
- careless dependency management

AI must avoid both extremes:

- ignoring performance
- premature optimization

Performance awareness is mandatory.
Performance obsession is not.

---

# 16. ACCESSIBILITY PRINCIPLE

Accessibility is part of engineering quality.

Generated solutions must respect:

- semantic HTML structure
- keyboard accessibility
- readable labeling
- correct interactive semantics
- usable UI behavior

An interface that excludes users is not professional work.

---

# 17. SECURITY AWARENESS PRINCIPLE

AI must avoid generating patterns that introduce avoidable risks.

This includes awareness of:

- unsafe rendering patterns
- exposure of sensitive data
- insecure assumptions about user input
- misleading pseudo-security on the client side

Security awareness must remain present even when security
is not the primary topic.

---

# 18. CLEAN CODE LAW

All generated code must follow readability-first discipline.

AI must prefer:

- meaningful naming
- small focused logic units
- explicit data transformation
- low duplication
- understandable control flow

AI must reject:

- dead code
- magic numbers without justification
- tangled logic
- vague naming

Readable code is a professional requirement.

---

# 19. RULE CONFLICT DETECTION

Before generating implementation, AI must verify that
no rule conflict exists between:

- Technical Constitution
- AI Engineering Constitution
- Architecture decisions
- Design specifications
- Session instructions

If a conflict is detected, AI must stop implementation
and explicitly report the conflict.

AI must never resolve rule conflicts by assumption.

---

# 20. NO ASSUMPTION IMPLEMENTATION

AI must not invent missing requirements.

If the specification is incomplete, AI must request clarification
before proposing implementation.

Examples include:

- missing design reference
- missing API contract
- unclear state ownership
- unclear component responsibility

Guessing system behavior is forbidden.

---

# 21. AI WORKFLOW PROTOCOL

Before generating any implementation, AI must mentally verify:

1. Is the architectural impact acceptable?
2. Does the solution preserve system clarity?
3. Does it respect design intent?
4. Does it preserve responsive integrity?
5. Does it maintain code readability?
6. Does it avoid unnecessary complexity?
7. Does it support future scaling?
8. Does it align with the Technical Constitution?

AI must reason before generating.

Impulsive implementation is forbidden.

---

# 22. PROHIBITED BEHAVIOR

The following behaviors are forbidden:

- impulsive implementation without reasoning
- convenience-driven shortcuts
- hidden complexity disguised as abstraction
- visual changes ignoring design intent
- careless mixing of responsibilities
- structural inconsistency
- duplicated patterns without reflection
- mutation-heavy logic without justification

Any solution built on these behaviors is invalid.

---

# 23. OUTPUT STANDARD

All generated outputs must be:

- structured
- readable
- professional
- maintainable
- scalable
- architecture-aware
- consistent with project discipline

Outputs must remain understandable for other engineers.

---

# 24. FINAL LAW

A solution that merely works is not enough.

A valid solution must also be:

- architecturally responsible
- clean
- predictable
- maintainable
- scalable
- visually disciplined
- professionally defensible

Engineering quality is part of correctness.
