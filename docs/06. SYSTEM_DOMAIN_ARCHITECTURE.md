# 2. ARCHITECTURAL MODEL OVERVIEW

The Social Media Web App is not modeled as one giant generic content system.

It is modeled as a set of bounded domains that interact in controlled ways.

The major system-level domains are:

1. Auth Domain
2. Self Profile Domain
3. Public Profile Domain
4. Feed Domain
5. Explore Post Domain
6. Post Domain
7. Comment Domain
8. Search Domain
9. Like Domain
10. Follow Domain
11. Save Domain

Important distinction:

- not all system-supported domains are active in Phase 1 frontend UI
- some domains exist in backend capability but are deferred in frontend interaction scope

This distinction must be preserved.

# 3. SYSTEM CAPABILITY VS PHASE 1 FRONTEND SCOPE

## 3.1 Full System Capability Surface

Based on the official API contract, the backend system currently supports:

- authentication
- self profile
- public profile
- explore posts
- authenticated feed
- post creation
- post detail
- comments
- likes
- follow system
- saved posts
- followers/following lists
- my liked posts
- user liked posts
- my posts
- user posts
- user search

## 3.2 Active Phase 1 Frontend Domains

Phase 1 frontend implementation actively focuses on:

- Auth Domain
- Self Profile Domain
- Public Profile Domain
- Feed Domain
- Post Domain
- Comment Domain
- Search Domain

The following domains may be read passively through API data fields
but are not mandatory Phase 1 interactive frontend domains:

- Like Domain
- Follow Domain
- Save Domain

Examples:

- a post card may display `likeCount`
- a profile result may include `isFollowedByMe`
- a post response may include `likedByMe`

However, this does NOT mean
Phase 1 frontend must implement full like/follow/save flows.

# 4. CORE DOMAIN ENTITIES

The system is built around several core entities and composed views.

Core system entities and surfaces:

1. User
2. Post
3. Comment
4. Feed
5. Search Result Surface
6. Relationship Surface

Important:

- not every domain corresponds to a database entity
- some domains are composed query surfaces
- some domains represent relationship behavior rather than standalone entities

---

# 5. ENTITY AND SURFACE DEFINITIONS

## 5.1 USER

Represents a platform user.

Typical backend-facing attributes may include:

- id
- username
- name
- email
- phone
- avatarUrl
- bio
- createdAt

User appears in multiple contexts:

- login response
- self profile
- public profile
- post author
- comment author
- search result
- follower/following list
- like list

User is a shared cross-domain entity,
but ownership of profile views depends on the domain context:

- self-owned profile access belongs to Self Profile Domain
- public-owned profile access belongs to Public Profile Domain

---

## 5.2 POST

Represents a piece of content created by a user.

Typical attributes may include:

- id
- imageUrl
- caption
- createdAt
- author
- likeCount
- commentCount
- likedByMe

Post appears in:

- explore posts
- feed/timeline
- post detail
- self posts
- public user posts
- saved posts
- liked posts

Post entity is owned by the Post Domain.

However, multiple composed domains may consume post data
without owning post creation or post deletion behavior.

---

## 5.3 COMMENT

Represents a comment attached to a post.

Typical attributes may include:

- id
- postId
- author
- text
- createdAt

Comments always belong to a post surface.

Comment entity is owned by the Comment Domain.

Comments may be displayed inside Post Domain surfaces,
but comment querying and comment mutation logic remain
owned by Comment Domain.

---

## 5.4 FEED

Feed is not a standalone database entity.

Feed is a composed query surface.

It represents:

- authenticated timeline posts
- self + following post stream
- ordered collection of post previews

Feed is owned by the Feed Domain.

Feed consumes post data.
Feed does not own post creation.
Feed does not own post deletion.

Feed must not be confused with explore posts.

---

## 5.5 SEARCH RESULT SURFACE

Search is not a database entity.

Search is a query surface that returns domain-owned entities.

In Phase 1 frontend,
the active required search surface is:

- user search results

Search consumes data from:

- User-related profile domains

Search does not own users.
Search does not own posts.
Search only orchestrates result retrieval and presentation.

---

## 5.6 RELATIONSHIP SURFACE

Some backend responses expose relationship signals between users or users and posts.

Examples:

- likedByMe
- isFollowedByMe
- followsMe
- isMe

These are not independent Phase 1 UI domains by default.

They are relationship metadata surfaces
consumed by UI when provided by the backend.

Relationship behavior may later expand into:

- Like Domain UI
- Follow Domain UI
- Save Domain UI

but those interaction flows are not mandatory in Phase 1 frontend scope.

# 6. DOMAIN OWNERSHIP MODEL

Each domain owns its own logic, query responsibilities,
mutation responsibilities, and UI composition boundaries.

| Domain                | Owns                                                                 |
| --------------------- | -------------------------------------------------------------------- |
| Auth Domain           | login, register, auth submission, auth session entry flow            |
| Self Profile Domain   | authenticated profile view, profile edit, my posts/related self data |
| Public Profile Domain | public user profile view, public user post-related views             |
| Feed Domain           | authenticated timeline feed composition                              |
| Explore Post Domain   | public explore post listing surface                                  |
| Post Domain           | post creation, post detail, post deletion ownership                  |
| Comment Domain        | comment list, add comment, delete comment behavior                   |
| Search Domain         | user search flow and search result orchestration                     |
| Like Domain           | like/unlike behavior and like-list behavior                          |
| Follow Domain         | follow/unfollow and followers/following behavior                     |
| Save Domain           | save/unsave and saved-post behavior                                  |

Important:

- a domain may consume another domain’s entity
- consuming an entity does not transfer ownership
- displaying related data does not equal owning that logic

---

# 7. DOMAIN RESPONSIBILITIES

## 7.1 AUTH DOMAIN

Responsible for:

- login form
- register form
- validation and submission
- session entry flow
- token acquisition
- auth-bound route access orchestration

Primary routes:

/auth/login  
/auth/register

Primary API alignment:

- POST /api/auth/register
- POST /api/auth/login

Auth Domain does not own profile query refresh.
Authenticated profile truth belongs to Self Profile Domain.

---

## 7.2 SELF PROFILE DOMAIN

Responsible for:

- authenticated self profile view
- edit profile flow
- self-owned profile data refresh
- my own profile surface
- my own posts and related self data surfaces when needed

Primary routes:

/profile  
/profile/edit

Primary API alignment:

- GET /api/me
- PATCH /api/me
- GET /api/me/posts
- GET /api/me/likes
- GET /api/me/followers
- GET /api/me/following
- GET /api/me/saved

Important:

- `/api/me` is the self profile truth
- this domain must not be modeled as public profile reuse by shortcut

---

## 7.3 PUBLIC PROFILE DOMAIN

Responsible for:

- viewing another user’s profile
- viewing public user-owned surfaces
- rendering public user identity and public relationship surfaces

Primary route:

/users/[username]

Primary API alignment:

- GET /api/users/{username}
- GET /api/users/{username}/posts
- GET /api/users/{username}/likes
- GET /api/users/{username}/followers
- GET /api/users/{username}/following

Important:

- public profile uses `username` path ownership
- public profile is distinct from self profile
- self profile and public profile must not share cache meaning carelessly

---

## 7.4 FEED DOMAIN

Responsible for:

- authenticated timeline feed
- feed list orchestration
- infinite scroll or paginated feed consumption
- loading/empty/error state for authenticated timeline

Primary route:

/timeline

Primary API alignment:

- GET /api/feed

Important:

- feed = authenticated self + following timeline
- feed is not explore posts
- feed consumes post preview data but does not own post creation

---

## 7.5 EXPLORE POST DOMAIN

Responsible for:

- public explore listing surface
- public post browsing surface when required by feature composition

Primary API alignment:

- GET /api/posts

Important:

- this is a system-supported domain
- Phase 1 frontend may consume explore-style post data where necessary
- this must not be confused with feed/timeline ownership

If a dedicated explore page is added later,
it belongs here, not in Feed Domain.

---

## 7.6 POST DOMAIN

Responsible for:

- create post
- post detail view
- post deletion ownership
- post model ownership
- post-level UI orchestration

Primary routes:

/posts/create  
/posts/[postId]

Primary API alignment:

- POST /api/posts
- GET /api/posts/{id}
- DELETE /api/posts/{id}

Important:

- create post belongs here, not in Feed Domain
- post detail belongs here, though it may render Comment Domain surfaces inside it
- post domain owns post lifecycle behavior

---

## 7.7 COMMENT DOMAIN

Responsible for:

- comment list querying
- add comment mutation
- delete comment behavior
- comment-level loading/empty/error handling

Primary route surface:

/posts/[postId]

Primary API alignment:

- GET /api/posts/{id}/comments
- POST /api/posts/{id}/comments
- DELETE /api/comments/{id}

Important:

- comments belong to posts relationally
- but comment query/mutation logic is still owned by Comment Domain
- Post Domain may host Comment UI surface, but not swallow Comment Domain ownership

---

## 7.8 SEARCH DOMAIN

Responsible for:

- search input orchestration
- query-state handling for search
- search result rendering
- search loading/empty/error state

Primary routes:

/search  
/search/users

Primary API alignment:

- GET /api/users/search

Phase 1 note:

- user search is active
- post search is not required in Phase 1 frontend roadmap

Search Domain consumes domain data.
It does not own User or Post entities.

---

## 7.9 LIKE DOMAIN

Responsible at system level for:

- like a post
- unlike a post
- list users who liked a post
- my liked posts
- public user liked posts

Primary API alignment:

- POST /api/posts/{id}/like
- DELETE /api/posts/{id}/like
- GET /api/posts/{id}/likes
- GET /api/me/likes
- GET /api/users/{username}/likes

Phase 1 rule:

- this domain exists in system capability
- full interactive Like UI is not mandatory in Phase 1 frontend

However, backend like-related fields may still appear in UI surfaces.

---

## 7.10 FOLLOW DOMAIN

Responsible at system level for:

- follow user
- unfollow user
- followers list
- following list
- relationship flags involving follow behavior

Primary API alignment:

- POST /api/follow/{username}
- DELETE /api/follow/{username}
- GET /api/me/followers
- GET /api/me/following
- GET /api/users/{username}/followers
- GET /api/users/{username}/following

Phase 1 rule:

- this domain exists in system capability
- full follow-management UI is not mandatory in Phase 1 frontend

The system language is follow-based, not friend-based.

---

## 7.11 SAVE DOMAIN

Responsible at system level for:

- save a post
- unsave a post
- list my saved posts

Primary API alignment:

- POST /api/posts/{id}/save
- DELETE /api/posts/{id}/save
- GET /api/me/saved

Phase 1 rule:

- this domain exists in system capability
- bookmark/save management UI is not mandatory in Phase 1 frontend

# 8. ENTITY RELATIONSHIP MODEL

The system relationship structure can be read as follows.

User
├── creates → Post
├── writes → Comment
├── owns → Self Profile surface
├── appears in → Public Profile surface
├── appears in → Search results
├── may follow → User
├── may like → Post
└── may save → Post

Post
├── belongs to → User (author)
├── contains → Comments
├── appears in → Feed
├── appears in → Explore Posts
├── appears in → Profile post surfaces
├── may be liked by → User
└── may be saved by → User

Comment
├── belongs to → Post
└── belongs to → User

Feed
└── displays → Posts

Search
└── returns → Users (Phase 1 active)

Relationship Surface
├── connects → User ↔ User
└── connects → User ↔ Post

# 9. DOMAIN INTERACTION RULES

Domain interaction is allowed only through clear consumption boundaries.

Rules:

1. A domain may consume another domain’s data surface
   without taking over its ownership.

2. A domain must not silently mutate another domain’s responsibilities.

3. Post Domain may host Comment UI surface,
   but must not absorb comment query/mutation logic carelessly.

4. Feed Domain may render Post cards,
   but must not own post creation logic.

5. Search Domain may render User cards,
   but does not own User entity lifecycle.

6. Self Profile Domain and Public Profile Domain
   must remain separate in ownership even if they share UI primitives.

7. Like / Follow / Save metadata may appear in Post or Profile surfaces,
   but their system ownership remains in their own domains.

# 10. DATA FLOW MODEL

Typical user flows create the following domain interactions.

---

## 10.1 LOGIN FLOW

User
→ Auth Form
→ Auth Mutation
→ Auth API
→ Token acquired
→ Authenticated route access enabled
→ Self Profile may later refresh from /api/me

---

## 10.2 SELF PROFILE FLOW

Profile Page
→ Self Profile Query
→ Self Profile API
→ Render authenticated profile surface

Edit Profile Form
→ Profile Mutation
→ PATCH /api/me
→ Self profile invalidated/refreshed

---

## 10.3 PUBLIC PROFILE FLOW

Public User Page
→ Public Profile Query
→ Public User API
→ Render public user surface

Optional related tabs later may consume:

- public posts
- public likes
- public followers/following

---

## 10.4 FEED FLOW

Timeline Page
→ Feed Query
→ GET /api/feed
→ Render feed post cards
→ Paginate / append as required

---

## 10.5 CREATE POST FLOW

Create Post Form
→ Post Mutation
→ POST /api/posts
→ Post created
→ Feed invalidated/refreshed as required
→ Related post surfaces refreshed intentionally

---

## 10.6 POST DETAIL FLOW

Post Detail Page
→ Post Detail Query
→ GET /api/posts/{id}
→ Render post detail surface
→ Comment Domain surface mounted inside as needed

---

## 10.7 COMMENT FLOW

Post Detail Page
→ Comment List Query
→ GET /api/posts/{id}/comments
→ Render comments

Comment Form
→ Add Comment Mutation
→ POST /api/posts/{id}/comments
→ Comment list invalidated/refreshed

Delete Comment Action
→ DELETE /api/comments/{id}
→ Comment list invalidated/refreshed

---

## 10.8 SEARCH FLOW

Search Input
→ Search Query State
→ GET /api/users/search
→ Return user results
→ Render search results

# 11. STATE OWNERSHIP RULES

Server-backed domain data must live in TanStack Query.

Examples:

- self profile data
- public profile data
- feed data
- post detail data
- comment list data
- search result data
- my posts
- public user posts
- liked/saved/follower/following lists when those surfaces become active

Client UI state may live in:

- React local state
- Redux Toolkit for shared UI-only state

Forbidden:

- storing fetched entities in Redux
- blending server state and UI state carelessly
- using route/page files as ad hoc server-state orchestration dumps

---

# 12. QUERY OWNERSHIP MODEL

Query ownership should follow domain ownership.

Examples:

- Auth Domain owns auth mutation flows
- Self Profile Domain owns self profile queries
- Public Profile Domain owns public-profile queries
- Feed Domain owns feed queries
- Post Domain owns post detail and post creation/deletion flows
- Comment Domain owns comment list and comment mutations
- Search Domain owns user search queries
- Like Domain owns like-related queries/mutations when activated
- Follow Domain owns follow-related queries/mutations when activated
- Save Domain owns save-related queries/mutations when activated

This prevents query logic from scattering across unrelated files.

---

# 13. MUTATION OWNERSHIP MODEL

Mutation ownership must be explicit.

| Mutation Type          | Domain Owner        |
| ---------------------- | ------------------- |
| login                  | Auth Domain         |
| register               | Auth Domain         |
| update profile         | Self Profile Domain |
| create post            | Post Domain         |
| delete post            | Post Domain         |
| add comment            | Comment Domain      |
| delete comment         | Comment Domain      |
| like / unlike post     | Like Domain         |
| follow / unfollow user | Follow Domain       |
| save / unsave post     | Save Domain         |

Phase 1 note:

- some mutation domains exist structurally before their UI interaction is activated
- structural existence does not force immediate frontend implementation scope

---

# 14. DTO → DOMAIN MODEL FLOW

Raw backend responses must not be consumed carelessly by UI components.

Approved flow:

API DTO
→ domain mapper
→ domain model or view model
→ UI component

Examples:

- `UserDto` → `User`
- `PostDto` → `Post`
- `CommentDto` → `Comment`

If a UI needs a transformed structure,
a view model may be introduced intentionally.

---

# 15. DOMAIN BOUNDARY RULES

Each domain must control its own logic.

Examples:

- post creation logic must live inside Post Domain
- comment logic must live inside Comment Domain
- feed query logic must live inside Feed Domain
- public profile logic must live inside Public Profile Domain
- self profile logic must live inside Self Profile Domain

Forbidden patterns:

- Feed Domain owning post creation
- Post Domain silently owning search behavior
- Search Domain owning user profile data lifecycle
- route files swallowing all business logic
- self profile modeled as public profile shortcut
- public profile modeled as self profile shortcut
- renaming follow behavior into friends architecture without formal translation

---

# 16. PHASE 1 DOMAIN SAFETY RULES

Because Phase 1 is intentionally controlled,
the following rules apply:

1. Active frontend Phase 1 domains must remain limited to the official roadmap.

2. Domains such as Like / Follow / Save may exist structurally
   in architecture documents and API contract awareness,
   but their interactive UI flows are deferred unless roadmap changes.

3. UI may display backend-provided relationship metadata
   without forcing full interaction ownership into Phase 1.

4. No domain should be implemented merely because backend support exists.

System capability does not automatically equal Phase 1 priority.

---

# 17. FUTURE DOMAIN EXTENSIONS

Later phases may expand the architecture with dedicated domains such as:

- Notification Domain
- Messaging Domain
- Settings Domain
- Recommendation Domain
- Advanced Social Graph Domain

These must be introduced only through explicit roadmap revision.

---

# 18. ARCHITECTURAL SAFETY RULES

The following behaviors are forbidden:

- features directly calling random APIs
- mixing post and comment ownership carelessly
- feed owning post lifecycle logic
- search owning user or post entities
- Redux storing server data
- UI components depending on raw DTO casually
- self profile and public profile treated as the same domain
- explore posts and feed treated as the same domain
- follow system renamed into friends by careless abstraction
- domain logic leaking across features without explicit boundary

Violating these rules leads to architecture instability.

All features must respect:

- domain ownership
