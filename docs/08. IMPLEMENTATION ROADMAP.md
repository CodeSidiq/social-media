# 8. PHASE 7 — SELF PROFILE

Focus: Authenticated self profile

## Session 7A — Self Profile Header + Stats

Route:

- /profile

Endpoint:
GET /api/me

Domain Ownership:

- Self Profile Domain (NOT Auth Domain)

Primary Goal:
Bind authenticated self profile data into a dedicated self profile header surface.

Scope:

- avatar
- name
- username
- bio section
- quick stats (postsCount, followersCount, followingCount if available)
- edit profile CTA

Tasks:

- implement route /profile
- create self profile page container (client boundary only where required)
- integrate GET /api/me via query hook
- render self profile header identity
- render bio section
- render quick stats section
- render edit profile CTA
- ensure loading / empty / error state are explicitly handled

Rules:

- must NOT depend semantically on Auth Domain ownership
- query ownership must be moved or aligned to Self Profile Domain
- must NOT assume GET /api/me response shape without verification
- must follow service → query hook → UI data flow
- must NOT mix Self Profile Domain with Public Profile Domain
- must NOT include My Posts surface in this session
- must NOT introduce follow / like / bookmark interaction

Architecture Notes:

- current implementation may still reuse authApi.getMe temporarily
  but ownership MUST be transitioned to profile domain semantics

- query key must evolve from:
  queryKeys.auth.me
  → to:
  queryKeys.profile.me

- DTO must be unified:
  avoid dual meaning between:
  MeResponseDto
  and
  MeProfilePayloadDto

  one endpoint = one contract

- if backend returns:
  profile + stats
  then mapper must normalize into a clean domain model

Exit checkpoint:

- self profile header renders correctly
- data is sourced from GET /api/me
- domain ownership is respected
- no leakage into public profile domain
- no assumption-based field usage
- UI is stable across loading / error / success states

## Session 7B — My Posts Surface

Endpoint:
GET /api/me/posts

Tasks:

- self posts surface
- empty state
- create-first-post CTA if appropriate

Rule:

Use the term "Self Profile" or "My Profile".
Do NOT invent "friend profile" terminology.
The system is follow-based, not friend-based.

Exit checkpoint:

- self profile works
- self posts surface works
- self vs public profile boundary remains clear

# 14. IMPORTANT WARNINGS

Warning 1

confirmPassword does NOT exist in backend.
It must remain frontend-only validation.

Warning 2

The payload for:

- GET /api/me
- GET /api/users/{username}

must not be finalized by assumption.
Use verified contract awareness only.

Warning 3

Feed has two distinct data surfaces:

- Explore:
  - GET /api/posts
- Timeline:
  - GET /api/feed

Do NOT merge them carelessly.

Warning 4

Like / Save / Follow / Followers / Following
exist in backend capability,
but remain deferred until officially activated.

Warning 5

Add Post and Update Profile use:

- multipart/form-data

NOT JSON.

Form handling,
DTO handling,
and request construction
must respect that difference explicitly.

Warning 6

Naming must remain domain-accurate.

Use:

- self profile
- public profile
- follow
- followers
- following
- feed
- explore

Do NOT invent:

- friends profile
- friends list
- friend relationship

unless the architecture is formally revised.

Warning 7

This session plan must never be treated as a rival roadmap.
It is an execution document only.

If any future patch introduces ordering conflicts,
the conflicting patch must be rejected
until this document is updated explicitly.
