# 4. ARCHITECTURE PRINCIPLES

All implementation must follow these principles.

## 4.1 Predictability First

Code must be easy to trace, reason about, and debug.

## 4.2 Single Responsibility

Each module, component, hook, and function must have one clear responsibility.

## 4.3 Separation of Concerns

UI, server state, client state, validation, data fetching,
and business logic must not be mixed carelessly.

## 4.4 Reusability with Discipline

Reusable code must emerge from real repeated need,
not from premature abstraction.

## 4.5 Feature-Oriented Scalability

The codebase must support feature growth
without collapsing into shared-folder chaos.

## 4.6 Maintainability Over Cleverness

Readable, stable, explicit code is preferred over compact but obscure code.

## 4.7 No Hidden Coupling

No feature may silently depend on unstable assumptions from another feature.

# 5. NEXT.JS APP ROUTER RULES

## 5.1 App Router is mandatory

This project must use Next.js App Router.

## 5.2 Default to Server Components

All components must be treated as Server Components by default
unless a Client Component is actually required.

## 5.3 Client Components are allowed only when necessary

Use `"use client"` only when the component needs one or more of the following:

- browser-only APIs
- local interactive state
- event handlers
- TanStack Query hooks
- Redux hooks
- React Hook Form behavior
- Framer Motion client behavior
- other client-only libraries

## 5.4 Minimize client boundary spread

Do not promote large subtrees to Client Components
unless truly necessary.

## 5.5 No unnecessary client wrappers

Do not wrap static presentational content
in Client Components without clear technical need.

# 6. TYPESCRIPT LAW

## 6.1 Strict typing is mandatory

TypeScript must be used in strict mode.

## 6.2 No `any`

`any` is prohibited unless there is a formally justified exceptional case.

Allowed alternatives:

- `unknown`
- generics
- properly inferred types
- explicit interfaces/types
- zod inference

## 6.3 Real contract alignment

Types must reflect real backend/API contracts,
not imaginary idealized shapes.

## 6.4 Type duplication should be minimized

When a schema or DTO shape already defines the contract,
prefer deriving types rather than rewriting identical manual types.

## 6.5 Prefer explicit return types where clarity matters

Return types should be explicit when the function is:

- non-trivial
- shared
- exported
- critical for readability

# 7. STATE MANAGEMENT BOUNDARIES

State ownership is strictly divided.

## 7.1 Redux Toolkit = client UI state only

Redux Toolkit may be used only for client-side UI state, for example:

- modal visibility
- sidebar state
- drawer state
- selected tab
- local workflow coordination state
- filter UI state
- ephemeral interface control state

Redux Toolkit must NOT be used for:

- API response caching
- fetched lists
- remote entities
- mutation payload persistence
- duplicated server resource storage

## 7.2 TanStack Query = server state only

TanStack Query must handle:

- data fetching
- caching
- background refetching
- invalidation
- pagination / infinite queries
- mutation coordination with server-backed data

## 7.3 React local state = local component interaction

Use React local state only for isolated UI behavior
that does not require app-level sharing.

## 7.4 State meaning must remain explicit

Every stateful decision must answer clearly:

- who owns the state
- why the state exists
- who consumes it
- when it resets
- whether it is local, shared UI, or server-backed

---

# 8. DATA FLOW LAW

Approved server data flow:

Axios Client → Service / API Layer → Query Hook / Mutation Hook → UI Component

Rules:

- Components must not call remote APIs directly
- Components must not manually transform raw backend contracts inline
  if that transformation belongs to service/query layer
- Query hooks must expose clean, predictable outputs
- Business-level mapping must be centralized,
  not repeated across UI files

Forbidden patterns:

- fetch/axios directly inside random components
- ad hoc API logic inside JSX/TSX route files
- duplicating server transformation logic in multiple components
- storing fetched server entities in Redux

---

# 9. AXIOS / API LAYER RULES

## 9.1 Centralized client

All HTTP requests must go through the approved Axios client.

## 9.2 No raw scattered API calls

No feature may instantiate its own ad hoc HTTP layer
without architectural reason.

## 9.3 API contracts and DTOs

Raw backend response shapes must be treated as DTOs
when appropriate.

Rules:

- DTO types may remain close to service/API layer
- UI components should consume cleaned domain/view shapes
  when raw DTOs are unsuitable
- Mapping from DTO → Domain Model must happen in service/query layer,
  not inside presentational JSX

## 9.4 No silent response assumptions

Every endpoint integration must follow the official backend contract exactly.

If the contract is unclear, implementation must pause until clarified.

## 9.5 API contract authority

The official source of truth for integration details is:

- API_INTEGRATION_CONTRACT.md

This includes:

- endpoint path
- method
- auth requirement
- request shape
- response envelope
- pagination format
- field naming conventions

---

# 10. QUERY RULES

## 10.1 TanStack Query ownership

All server state hooks must be implemented with TanStack Query.

## 10.2 Query keys must be consistent

Query keys must be centralized through query key factories.

Approved pattern examples:

- `feedKeys.all`
- `feedKeys.list(params)`
- `postKeys.all`
- `postKeys.detail(postId)`
- `commentKeys.list(postId)`
- `profileKeys.me()`
- `profileKeys.public(username)`
- `searchKeys.users(params)`

## 10.3 No inconsistent string scattering

Do not hardcode unrelated random query keys across files.

## 10.4 Query hooks should be feature-scoped

Query hooks should live near the feature they belong to
unless they are intentionally shared across multiple domains.

## 10.5 Mutation handling must be explicit

Each mutation must define:

- success handling
- error handling
- invalidation/refetch behavior
- optimistic update strategy if applicable
- rollback strategy if optimistic update is used

## 10.6 Endpoint meaning must affect cache separation

The following must not share cache meaning carelessly:

- explore posts vs timeline feed
- self profile vs public user profile
- my posts vs public user posts
- my liked posts vs public user liked posts
- saved posts vs liked posts

# 11. FORM RULES

## 11.1 React Hook Form + Zod are the standard

Forms must use:

- React Hook Form for form state
- Zod for schema validation

## 11.2 Validation schema placement

Rules:

- feature-specific form schemas belong inside that feature
- shared reusable schemas may live in shared schema folders
  only if reused by multiple features

## 11.3 Prefer schema-derived typing

Prefer `z.infer<typeof schema>` when appropriate
to avoid duplicated type declarations.

## 11.4 Form UI and submission logic must be cleanly separated

Form rendering, validation, submission,
and post-submit handling must not become tangled
inside one monolithic component.

## 11.5 Multipart forms must obey API contract

Any form involving file upload must follow the exact API contract.

Examples:

- profile update form must use the backend-approved multipart fields
- create post form must use the backend-approved image field name

No form may invent upload payload structure.

# 12. FOLDER STRUCTURE LAW

The codebase must follow a clear and scalable structure.

Approved top-level source structure:

- `src/app`
- `src/components`
- `src/features`
- `src/lib`
- `src/services`
- `src/store`
- `src/types`
- `src/constants`
- `src/config`

Detailed meaning:

## 12.1 `src/app`

Contains:

- routes
- route segments
- layout files
- page files
- route-level loading/error boundaries

Must NOT become a dumping ground
for unrelated business logic.

## 12.2 `src/components`

Contains shared components only.

Examples:

- layout components
- shared UI wrappers
- cross-feature presentational components
- global providers if needed

Sub-rules:

- reusable primitive UI components belong in `src/components/ui`
- shared layout components belong in `src/components/layout`
- truly shared display components may live in other clear subfolders

## 12.3 `src/features`

Contains feature-oriented modules.

Each feature may contain only what belongs to that feature.

Recommended feature substructure:

- `components`
- `hooks`
- `services` or `api`
- `schemas`
- `types`
- `utils`
- `constants`

A feature should own its local logic
as much as possible before escalating to shared folders.

## 12.4 `src/lib`

Contains cross-cutting technical utilities.

Examples:

- helper functions
- formatters
- utility functions
- shared technical adapters
- shared schema helpers
- shared query helpers

## 12.5 `src/services`

Contains global service-level infrastructure
when it is not feature-local.

Examples:

- axios instance
- interceptors
- global API helpers

## 12.6 `src/store`

Contains Redux store setup, slices,
selectors, and store-related shared utilities.

## 12.7 `src/types`

Contains truly shared global types only.

Do not move feature-local types here
just because the folder exists.

## 12.8 `src/constants`

Contains truly shared constants only.

Do not pollute this folder with feature-local constants
that belong inside a feature.

## 12.9 `src/config`

Contains project-level configuration surfaces
that do not belong elsewhere.

Examples:

- environment helpers
- runtime config wrappers
- app-level static config

# 13. PLACEMENT RULES

To remove ambiguity, these rules are locked.

## 13.1 Shared vs feature-local

If a module is used by only one feature,
it must stay inside that feature
unless there is a strong shared need.

## 13.2 No fake reusability

Do not move code into shared folders prematurely.

## 13.3 UI primitives

Reusable primitive UI components must live in `src/components/ui`.

## 13.4 Feature forms

Feature forms and their schemas should stay inside their feature
unless intentionally shared.

## 13.5 Global services

Axios instance, interceptors, and global request infrastructure
belong in shared service infrastructure,
not inside unrelated features.

## 13.6 App routes are entry points, not logic warehouses

Route files should compose feature modules.
They must not become the primary home
for data orchestration and business logic.

## 14.2 Design token authority

Design tokens must originate from the official design system source.

Forbidden unless explicitly justified:

- raw hex color literals in TSX/JSX

Prefer semantic design-system classes
over literal styling when such classes exist.

# 18. HOOK LAW

## 18.1 Custom hooks must encapsulate repeatable logic

Hooks should exist to encapsulate repeatable behavior

A hook’s input, output, and side effects must be explicit.

## 18.3 No business logic duplication

# 19. ERROR HANDLING STANDARD

## 19.1 Error handling is mandatory

All remote operations must have defined error handling behavior.

## 19.2 Error UI pattern must be intentional

Error presentation may use:

- inline error message
- field-level form error
- toast notification
- retry UI
- page fallback UI
- section fallback UI

The chosen pattern must match the context.

## 19.3 Status-specific handling

The following status groups must not be treated identically
when the distinction matters:

- 400 validation/business errors
- 401 unauthenticated
- 403 unauthorized/forbidden
- 404 not found
- 409 conflict
- 500+ server failure

## 19.4 No swallowed errors

Errors must not disappear silently.

## 19.5 API-envelope awareness

Because backend failures use the response envelope:

- `success: false`
- `message`
- `data: null`

frontend error handling must not assume an `error` object exists.

# 34. DESIGN TOKEN IMPLEMENTATION CONTRACT

The Design System Foundations document defines
the official visual tokens of the system.

However, the implementation of those tokens inside the codebase
must follow a strict technical contract.

This prevents divergence between design and engineering.

Rules:

1. All visual tokens must originate from the official Design System Foundations document.

2. Raw tokens must be implemented as CSS variables inside the global styling layer.

Approved location example:

`src/app/globals.css`

3. CSS variable naming should follow a consistent token scheme.

Recommended prefixes:

- `--color-*`
- `--space-*`
- `--radius-*`
- `--font-*`

4. Components must not directly use raw design values.

Forbidden examples:

- `bg-[#F3F4F6]`
- `text-[#222]`
- `style={{ color: "#333" }}`

Correct approach:

Use semantic tokens or design-system utilities.

Examples:

- `bg-background`
- `text-foreground`

5. Tailwind arbitrary values must be minimized.

Arbitrary values such as:

- `w-[37px]`
- `top-[13px]`

may only be used when:

- design requires it
- no token or utility exists
- the value is not reusable

6. No parallel token system may be introduced.

All styling tokens must originate from
the official design token system.

---

# 35. SEMANTIC TOKEN LAYER

Between raw tokens and component usage,
a semantic layer must exist.

Raw tokens represent visual values.

Semantic tokens represent UI meaning.

Example mapping:

Raw token:
`--color-neutral-100`

Semantic token:
`--color-background`

Component usage:
`bg-background`

Example semantic tokens:

- `--color-background`
- `--color-foreground`
- `--color-card`
- `--color-border`
- `--color-muted`
- `--color-accent`
- `--color-primary`
- `--color-danger`

Components must prefer semantic tokens
rather than raw color tokens.

This ensures UI meaning remains stable
even if raw tokens change.

# 38. DATA MODEL NAMING CONVENTION

To prevent confusion between backend responses,
domain models, and UI-facing shapes,
the following naming convention is mandatory.

## 38.1 DTO (Data Transfer Object)

Represents raw backend response contracts.

Examples:

- `UserDto`
- `PostDto`
- `CommentDto`

DTO types must match backend API contracts exactly.

## 38.2 Domain Model

Represents the cleaned internal model
used by the application.

Examples:

- `User`
- `Post`
- `Comment`

Mapping from DTO → Domain Model must happen
in service or query layer.

## 38.3 View Model (optional)

Used only if UI requires a transformed data structure.

Examples:

- `PostViewModel`
- `UserProfileViewModel`

View models should exist only when necessary.

## 38.4 UI components should not consume DTO casually

UI components should consume domain/view models
unless there is a clear justified reason not to.

## 38.5 DTO naming must preserve backend field names

Fields such as the following must remain unchanged in DTO level:

- `avatarUrl`
- `createdAt`
- `likeCount`
- `commentCount`
- `likedByMe`
- `isFollowedByMe`
