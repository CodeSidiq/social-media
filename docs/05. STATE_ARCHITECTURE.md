# 0. PURPOSE

The purpose of this document is to prevent:

- server state chaos
- duplicated source of truth
- Redux misuse
- query cache abuse
- unnecessary re-renders
- invalid state ownership
- unpredictable mutation side effects
- careless state duplication across architectural layers

This document defines:

- what belongs to TanStack Query
- what belongs to Redux Toolkit
- what belongs to local component state
- what belongs to form state
- how invalidation must work
- how cache boundaries must be maintained
- how state decisions must align with domain ownership

This document does not redefine implementation law in general.

Implementation law belongs to:

- TECHNICAL_CONSTITUTION.md

- API_INTEGRATION_CONTRACT.md

Domain ownership truth belongs to:

- SYSTEM_DOMAIN_ARCHITECTURE.md

# 1. CORE STATE PRINCIPLES

All state must follow these principles:

1. Single source of truth
2. Clear ownership
3. Minimal duplication
4. Predictable updates
5. Separation of responsibilities
6. Maintainability over convenience
7. No assumption-based state design
8. No storing remote entities in the wrong layer
9. No fake synchronization between state systems
10. No hidden state side effects

State architecture must remain explicit.

If a state location cannot be justified,
it is probably wrong.

---

# 2. STATE CATEGORIES

The application uses four official state categories:

1. Server State
2. Global Client UI State
3. Local Component State
4. Form State

Each category has a different owner.

No state may be placed by habit or convenience alone.

---

# 3. SERVER STATE

## 3.1 Definition

Server state is data that originates from the backend
or is governed by backend truth.

Examples:

- authenticated self profile data
- public user profile data
- feed/timeline posts
- explore post lists
- post detail
- comment list
- search users results
- my posts
- public user posts
- liked/saved/followers/following lists when such surfaces are active

## 3.2 Owner

Server state belongs to:

- TanStack Query

## 3.3 Rules

Server state must:

- be fetched through approved API services
- be cached through TanStack Query
- be invalidated intentionally
- not be duplicated into Redux
- not be mirrored into random local state without strong reason
- not be treated as mutable UI-owned state

## 3.4 Phase 1 Active Examples

Server state examples in Phase 1 frontend:

Auth / Self Profile

- self profile data from `/api/me`

Public Profile

- public user profile data from `/api/users/{username}`

Feed

- timeline feed from `/api/feed`

Post

- post detail from `/api/posts/{id}`
- post creation mutation result handling

Comment

- comments list for a post

Search

- user search results from `/api/users/search`

Optional passive metadata may also appear in server state payloads:

- `likeCount`
- `commentCount`
- `likedByMe`
- `isFollowedByMe`

These fields may be consumed without activating full Like/Follow UI ownership.

## 3.5 Forbidden

The following are forbidden:

- storing fetched posts in Redux
- storing fetched comments in Redux
- storing fetched profiles in Redux
- manually syncing query data into Redux just for display
- using component state as a second cache for server state
- copying query data into local state just to “make rendering easier”

# 4. GLOBAL CLIENT UI STATE

## 4.1 Definition

Global client UI state is state that is:

- purely client-side
- not owned by the backend
- needed across multiple components or routes
- not just temporary local interaction

## 4.2 Owner

Global client UI state belongs to:

- Redux Toolkit

## 4.3 Allowed Examples

Examples of allowed Redux state:

- sidebar open/closed state
- active mobile navigation state
- global dialog visibility if truly global
- global toast orchestration if architecture justifies it
- cross-route UI filter shell state if truly client-owned
- shared layout display preferences if truly global

## 4.4 Rules

Redux must only store UI-oriented client state.

Redux must NOT become:

- remote entity storage
- API response warehouse
- duplicate cache layer
- form state manager
- random event bus
- hidden workflow dump

## 4.5 Forbidden

Do not store in Redux:

- feed posts
- explore posts response
- self profile API data
- public profile API data
- comments from backend
- search results from backend
- post detail response
- pagination data from backend
- mutation response data already owned by TanStack Query

---

# 5. LOCAL COMPONENT STATE

## 5.1 Definition

Local component state is temporary UI state
that is only needed inside one component
or a very small component subtree.

## 5.2 Owner

Local component state belongs to:

- React component state
- `useState`
- `useReducer` when justified locally

## 5.3 Allowed Examples

Examples:

- dropdown open/close state
- show/hide password toggle
- active carousel item
- local tab switching inside one component
- hover or expanded-card state
- local preview visibility
- page-scoped modal open/close state
- intersection observer helper flags for one screen section

## 5.4 Rules

Use local component state when:

- the state is temporary
- the state is UI-only
- the state is not needed globally
- the state does not require persistence across routes
- the state is not server-owned

Do not escalate local state into Redux without strong reason.

## 5.5 Forbidden

The following are invalid uses:

- putting one-component toggle state into Redux
- copying fetched query data into local state for normal rendering
- using local state as hidden global coordination
- using local state to simulate cache ownership

---

# 6. FORM STATE

## 6.1 Definition

Form state is input-driven user-editing state.

Examples:

- login form input values
- register form input values
- create post form values
- update profile form values
- add comment form values

## 6.2 Owner

Form state belongs to:

- React Hook Form

Validation belongs to:

- Zod

## 6.3 Rules

Forms must use:

- React Hook Form for form state handling
- Zod for validation schema

Form state must remain separate from:

- Redux
- TanStack Query cache
- unrelated local UI state

## 6.4 Phase 1 Form Examples

Forms in Phase 1:

- login form
- register form
- update profile form
- create post form
- add comment form

## 6.5 Forbidden

The following are forbidden:

- storing form field values in Redux
- using TanStack Query as form storage
- validating forms ad hoc without approved schema flow
- mixing server state directly into mutable form state without proper default mapping
- inventing form payload shape that differs from the API contract

# 8. QUERY OWNERSHIP RULES

## 8.1 Query Responsibility

TanStack Query is responsible for:

- remote data fetching
- caching
- background refetching
- pagination / infinite queries
- query status
- stale/fresh lifecycle
- retry behavior if configured
- invalidation triggers

## 8.2 Query Placement

Query hooks must live in appropriate feature boundaries.

Examples:

- `features/auth/hooks`
- `features/profile/hooks`
- `features/feed/hooks`
- `features/post/hooks`
- `features/comment/hooks`
- `features/search/hooks`

Shared query logic may only move to shared folders
if it is truly cross-feature and justified.

## 8.3 Query Keys

Query keys must be:

- centralized
- predictable
- composable
- domain-aware

Example categories:

- auth
- profile
- feed
- posts
- comments
- search

## 8.4 Query Meaning Separation

The following queries must not share cache meaning carelessly:

- self profile vs public profile
- feed vs explore posts
- my posts vs public user posts
- my likes vs public user likes
- saved posts vs liked posts

## 8.5 Forbidden Query Behavior

The following are forbidden:

- ad hoc query key strings scattered everywhere
- fetching directly in presentational UI components without approved hook/service layer
- mixing unrelated domain data under the same query key
- query keys that are vague or unstable

---

# 9. MUTATION OWNERSHIP RULES

## 9.1 Mutation Responsibility

Mutations are responsible for changing backend data.

Examples:

- login
- register
- create post
- update profile
- add comment
- delete post
- delete comment

System-supported but Phase 1 optional interaction mutations include:

- like / unlike post
- follow / unfollow user
- save / unsave post

## 9.2 Mutation Placement

Mutation hooks must live inside the owning feature/domain.

Examples:

- `features/auth/hooks`
- `features/profile/hooks`
- `features/post/hooks`
- `features/comment/hooks`

## 9.3 Mutation Side Effects

Mutation side effects must be explicit.

Allowed examples:

- invalidate feed query after post creation
- invalidate comment list query after add comment
- invalidate self profile query after profile update
- redirect after successful login if defined
- reset form after successful creation if appropriate

## 9.4 Forbidden Mutation Behavior

The following are forbidden:

- hidden global side effects
- manual cache patching without strong reason
- random state updates across unrelated features
- mutation success handling split across too many layers
- using Redux to fake backend mutation success state when query invalidation should handle it

---

# 10. INVALIDATION STRATEGY

Invalidation must be intentional and minimal.

Do not invalidate everything blindly.

Invalidation rules for Phase 1:

## 10.1 Login / Register

After successful auth:

- store/use token according to auth implementation
- refresh self-profile query if needed
- redirect according to auth flow

## 10.2 Create Post

After successful post creation:

- invalidate feed-related queries
- invalidate post/explore-related queries only if the active UI depends on them
- optionally invalidate my-posts query if that surface is active

## 10.3 Add Comment

After successful comment creation:

- invalidate post-detail comment list
- invalidate post-detail query only if comment-dependent metadata must refresh

## 10.4 Update Profile

After successful profile update:

- invalidate self-profile query
- invalidate public-profile identity surfaces only if visible and affected
- invalidate other identity-bearing queries only when truly necessary

## 10.5 Search

Search generally does not require mutation invalidation in Phase 1.

## 10.6 Delete Post / Delete Comment

When these actions are active:

- invalidate only the affected post/feed/comment-related queries
- do not globally refetch unrelated domains

# 11. CACHE BOUNDARY RULES

Cache must respect domain boundaries.

Examples:

Self Profile cache:

- authenticated self profile data

Public Profile cache:

- public profile by username
- public user posts by username
- public user likes/followers/following when active

Feed cache:

- authenticated feed posts
- feed pagination metadata

Explore Post cache:

- public explore posts
- explore pagination metadata

Post Detail cache:

- one specific post detail

Comment cache:

- comments for one specific post

Search cache:

- user search results

Do not merge unrelated concerns into one cache record.

Each cache scope must stay understandable and domain-aligned.

---

# 12. OPTIMISTIC UPDATE POLICY

Optimistic updates are powerful but dangerous.

Phase 1 default rule:

- do NOT use optimistic updates unless clearly justified

Why:

- optimistic update adds cache complexity
- optimistic update increases rollback requirements
- optimistic update can corrupt mental model if designed carelessly

Optimistic updates may be considered later only if:

- mutation frequency is high
- UX benefit is meaningful
- rollback logic is fully defined
- ownership remains clear
- query cache boundaries remain safe

For Phase 1, prefer:

- mutation
- explicit success handling
- targeted invalidation
- refetch consistency

---

# 13. LOADING, ERROR, AND EMPTY STATE OWNERSHIP

## 13.1 Query State

Loading, error, and success states for server data belong to:

- TanStack Query consumption layer

## 13.2 UI Display

Visual rendering of those states belongs to:

- page container
- feature container
- reusable state component if appropriate

## 13.3 Rules

Do not create scattered custom status logic everywhere.

Do not hide query failure silently.

Do not render empty state as if it were success with zero explanation.

Every important query-driven screen must intentionally handle:

- loading
- error
- empty
- success

# 14. AUTH STATE ARCHITECTURE

Auth requires special handling.

## 14.1 Auth Data Categories

Auth-related state may include:

- auth form state
- session token awareness
- self-profile fetch state
- protected-route access behavior

## 14.2 Ownership

Auth form fields:

- React Hook Form

Auth session awareness:

- depends on approved auth implementation
- must follow known backend contract

Fetched current user / self-profile data:

- TanStack Query

UI-level auth visibility state:

- local state or Redux only if truly global and justified

## 14.3 Forbidden

- storing fetched current user profile in Redux by default
- mixing session storage logic randomly across features
- building auth assumptions without known backend contract

---

# 15. PROFILE STATE ARCHITECTURE

Profile state is split into two different surfaces.

## 15.1 Self Profile State

Includes:

- self profile query
- update profile mutation
- update profile form state
- self-owned profile sub-surfaces when active

Ownership:

- fetched self profile data → TanStack Query
- edit profile form → React Hook Form
- edit-only UI toggles → local state
- global profile-layout UI concerns → Redux only if truly global

## 15.2 Public Profile State

Includes:

- public profile query
- public user posts query when active
- public relationship-list surfaces when active

Ownership:

- fetched public profile data → TanStack Query
- page-local UI toggles → local state
- no Redux ownership unless the state is truly global UI-only

Important rule:

`/profile` and `/users/[username]` must remain separate route concerns
even if some UI primitives overlap.

---

# 16. FEED STATE ARCHITECTURE

Feed includes:

- authenticated feed query
- pagination / infinite loading state
- list rendering state
- item-local UI interaction state when needed

Ownership:

- feed data → TanStack Query
- page/cursor/pagination ownership → TanStack Query pagination model
- local intersection observer helper flags → local component state or ref
- route-wide UI display preferences → Redux only if justified

Important rule:

Feed owns feed consumption,
not post creation ownership,
and not explore-post ownership.

---

# 17. POST + COMMENT STATE ARCHITECTURE

## 17.1 Post Domain State

Owns:

- create post mutation
- post detail query
- delete post mutation when active

## 17.2 Comment Domain State

Owns:

- comments query for a post
- add comment mutation
- delete comment mutation when active

## 17.3 Ownership Rules

- post detail data → TanStack Query
- comments data → TanStack Query
- create post form → React Hook Form
- add comment form → React Hook Form
- local editor UI toggles → local state

Important rule:

Comments are child-domain behavior attached to post detail,
but Comment Domain still owns comment querying and mutation logic.

---

# 18. SEARCH STATE ARCHITECTURE

Search includes:

- search term handling
- result fetching
- result presentation state

Ownership:

- search results → TanStack Query
- route-driven category/surface selection → route state or local state
- persistent global search-shell preferences → Redux only if truly justified
- input control → local state or form state depending on implementation pattern

Phase 1 rule:

- active required search surface is user search
- post search is not mandatory in Phase 1 frontend scope

Important rule:

Search consumes User-related surfaces.
It does not own User entities.

---

# 19. SYSTEM-SUPPORTED BUT PHASE-DEFERRED STATE SURFACES

The system already supports some additional state surfaces through the backend,
even though Phase 1 does not require full interactive UI ownership.

Examples:

- like lists
- saved posts
- followers lists
- following lists
- my liked posts
- public user liked posts

Rules:

1. These may exist as server state surfaces in architecture.
2. Their existence does not force immediate Phase 1 UI implementation.
3. If later activated, they still belong to TanStack Query as server state.
4. Their UI toggles or view preferences remain separate from server-owned data.

---

# 20. FORBIDDEN STATE ANTI-PATTERNS

The following anti-patterns are invalid:

- fetched API data stored in Redux without strong reason
- same remote data duplicated in TanStack Query and Redux
- same form managed by `useState` and React Hook Form simultaneously without reason
- page state escalated to Redux prematurely
- optimistic update added without rollback plan
- mutation side effects hidden inside unrelated components
- DTO objects passed everywhere as UI truth without mapping discipline
- local state used as fake global coordination
- giant parent page component managing every state in one place
- self profile and public profile sharing one ambiguous cache identity
- feed and explore posts sharing one ambiguous cache identity

These anti-patterns must be rejected.

---

# 21. IMPLEMENTATION DECISION CHECKLIST

Before introducing any new state,
these questions must be answered:

1. Is this data from the backend?
2. Is this state global or local?
3. Is this state temporary or persistent across screens?
4. Is this editable user input?
5. Does TanStack Query already own this?
6. Would Redux duplicate a source of truth?
7. Is local state enough?
8. Does this require invalidation?
9. What is the side effect after mutation?
10. What would break if ownership is wrong?

If these questions cannot be answered clearly,
implementation must pause.

---

# 22. PHASE 1 STATE DECISION SUMMARY

Official Phase 1 default rules:

- remote entities → TanStack Query
- global UI-only state → Redux Toolkit if truly global
- small temporary UI state → local component state
- form input and validation → React Hook Form + Zod
- cache refresh → targeted invalidation
- optimistic update → avoid by default in Phase 1

This is the official baseline.

# 23. FINAL GOVERNING STATEMENT

This document exists to preserve state discipline.

No feature may introduce state casually.

Every state must have:

- a clear owner
- a justified location
- a predictable lifecycle
- a valid update pathway
- a domain-aligned responsibility

The system must not evolve into:

- duplicated truth
- cache confusion
- Redux misuse
- component-state spaghetti
- mutation side-effect chaos

If state architecture becomes unclear,
implementation must stop until clarified.
