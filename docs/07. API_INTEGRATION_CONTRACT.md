# SOCIAL MEDIA WEB APP

# API INTEGRATION CONTRACT

Version: 2.1
Last Updated: 14 March 2026
Owner: Tuan Sidiq

This document defines the official API integration contract between
Frontend and Backend systems for the Social Media Web App.

This document is based on the Swagger specification of the Sociality API.

The goal of this contract is to ensure:

- predictable API communication
- stable request/response handling
- consistent endpoint usage
- explicit authentication rules
- consistent pagination handling
- DTO awareness for frontend integration
- type-safe frontend implementation

Any backend API change that affects endpoint path,
HTTP method,
request shape,
response shape,
authentication requirement,
or pagination format
must update this document first.

---

# 0. GENERAL API RULES

All API integration must follow the real backend contract exactly.

No endpoint may be invented.
No request body may be assumed.
No response shape may be guessed.

If Swagger and actual backend behavior differ,
implementation must pause until clarified.

---

# 0.1 BASE URL

Official backend base URL:

https://be-social-media-api-production.up.railway.app

All endpoint paths in this document are relative to this base URL.

Example:

GET /api/posts

means:

GET https://be-social-media-api-production.up.railway.app/api/posts

---

# 0.2 GLOBAL RESPONSE ENVELOPE

The backend uses a unified response envelope.

Success pattern:

{
"success": true,
"message": "OK",
"data": {}
}

Failure pattern:

{
"success": false,
"message": "Human readable error message",
"data": null
}

Rules:

Frontend must not assume the message value.
Frontend must not assume error responses contain an `error` object.
Frontend must handle `data: null`.
Frontend must treat `success`, `message`, and `data` as the envelope.

---

# 0.3 AUTHORIZATION RULE

Protected endpoints must include:

Authorization: Bearer <token>

Rules:

Token is returned by login endpoint.
Protected routes must never be called without valid auth state.
401 responses must be handled intentionally.

---

# 0.4 CONTENT TYPE RULES

JSON endpoints:

Used for:

- register
- login
- add comment

Header:

Content-Type: application/json

Multipart endpoints:

Used for:

- update profile avatar
- create post with image

Header:

Content-Type: multipart/form-data

Rules:

Frontend must not send JSON to multipart endpoints.
FormData must use correct field names defined by backend.

---

# 0.5 PAGINATION RULE

Most list endpoints use:

?page=1&limit=20

Response format:

{
"success": true,
"message": "OK",
"data": {
"resourceArray": [],
"pagination": {
"page": 1,
"limit": 20,
"total": 0,
"totalPages": 0
}
}
}

Rules:

Frontend must read the resource key explicitly.
Pagination must use page, limit, total, totalPages.

---

# 0.6 NAMING CONVENTION RULE

Backend uses camelCase.

Examples:

avatarUrl
createdAt
likeCount
commentCount
totalPages
likedByMe
isFollowedByMe

Frontend DTO types must match backend naming exactly.

---

# 0.7 ID TYPE RULE

Resource identifiers:

user id integer
post id integer
comment id integer
username string

Rules:

Numeric ids must remain numbers.
Username must remain string.

# 1 AUTH API

Handles authentication.

Endpoints:

POST /api/auth/register
POST /api/auth/login

Authenticated user profile:

GET /api/me

---

## 1.1 REGISTER

POST /api/auth/register

Auth: Public

Request:

{
"name": "John Doe",
"username": "johndoe",
"email": "john@email.com",
"phone": "081234567890",
"password": "secret123"
}

Success:

HTTP 201

Failure example:

{
"success": false,
"message": "Phone already in use",
"data": null
}

Rules:

Frontend must send all required fields.

---

## 1.2 LOGIN

POST /api/auth/login

Auth: Public

Request:

{
"email": "john@email.com",
"password": "secret123"
}

Response:

{
"success": true,
"message": "Login success",
"data": {
"token": "JWT_TOKEN",
"user": { ... }
}
}

Rules:

Token must be stored securely.
Token is used for all protected endpoints.
